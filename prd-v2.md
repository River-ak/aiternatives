# PRD V2：AI工具替代方案引擎

> **文档状态**：V2（基于市场调研修正）
> **V2 变更摘要**：核心定位从"折扣+替代"翻面为"替代为主、折扣为辅" · 收入模型修正为两阶段漏斗 · 经常性佣金优先策略 · 新增Reddit/Twitter数据源
> **作者**：产品管理专家
> **日期**：2026-07-06

## V1 → V2 变更记录

| 章节 | V1 | V2 | 变更原因 |
|------|-----|-----|---------|
| **核心定位** | AI工具折扣聚合 + 替代方案对比 | **AI工具替代方案引擎**（折扣降级为辅助引流模块） | 调研发现折扣赛道有10+成熟玩家，差异化不成立 |
| **收入模型** | 单阶段漏斗 1.5-3% 总转化率 | 两阶段漏斗：CTR 2-4% × 试用转化 15-25% = **总 0.3-1.0%** | 调研发现真实漏斗必须分点击→试用→付费 |
| **佣金策略** | 无优先级区分 | **经常性佣金优先**（LTV高3.5倍） | 调研数据：20%经常性 vs 40%一次性，前者LTV高3.5x |
| **Agent数据源** | PH/HN/官网 | PH/HN/官网 + **Reddit r/AIToolsDeals + Twitter AI KOL** | 调研发现社区是优质折扣信息源 |
| **替代品数量** | 固定10个 | **5-15个，质量优先** | Grill Me压测：硬凑10个降低质量 |
| **12月收入目标** | $250/月 | **$300-800/月**（佣金策略决定上限） | 基于两阶段漏斗+经常性佣金复利效应重算 |

## 1. Problem Statement

### 用户端问题

**核心痛苦**：当一个AI工具涨价、停服、或功能不够用时，用户需要找到替代方案。现状极其低效——

- 在Google搜"ChatGPT alternatives"得到的信息是分散的、来自多个网站的列表文章，没有统一的对比维度
- 不同替代品的定价模式完全不同（按月/按年/终身/按用量），用户需要逐个打开官网手动对比
- 替代品推荐文章往往只是"作者用过的3-4个工具"，不是一个客观的、多维度的对比
- **用户花30分钟+在不同tab之间反复切换，才能做出一个选择**

### 为什么现在做

| 数据点 | 含义 |
|--------|------|
| AI SaaS 市场 $1,822亿（2026） | 替代方案的供给端在爆炸 |
| 64%新SaaS产品含AI功能 | 每个新工具都可能需要替代方案页 |
| AI SaaS仅占总SaaS的7% | 未来3-5年还有海量增长空间 |
| AlternativeTo 3.34页/访问、2分07秒停留 | 替代对比是极高参与度内容 |
| 91%公司使用AI SaaS工具 | 目标受众极其广泛 |

### 为什么我们

- 不做全品类软件导航（AlternativeTo已经做了），**只做AI工具替代方案**
- 不写主观推荐文章，**做结构化对比表**（价格/功能/评分/适合场景）
- Agent每天扫描新工具发布 → 自动发现替代关系 → 生成对比数据
- **每个替代方案页都是独立的SEO资产**，积累长尾价值

### 收入模型详解（V2修正版）

```
V1 的错误假设：
  400 PV × 3% CTR × 3% 转化率 × $50 = 完全错误（漏了试用转化阶段）

V2 修正版：
  阶段1：点击率（CTR）
    - 替代方案页（评测/对比属性）：2-4%
  
  阶段2：试用→付费转化率（发生在工具官网，非我们控制）
    - 低门槛AI工具（有免费试用，< $30/月）：15-25%
    - 中门槛AI工具（有免费试用，$30-100/月）：10-18%
  
  总漏斗 = CTR × 试用转化率
    = 2-4% × 10-25% = 0.2-1.0%

  一次性佣金模型（保守）：
  ┌─────────┬────────┬──────┬─────────┬──────────┬─────────┐
  │ 时间     │ 日均PV  │ CTR  │ 漏斗转化 │ 日均收入  │ 月收入   │
  ├─────────┼────────┼──────┼─────────┼──────────┼─────────┤
  │ 第3月    │ 300    │ 2.5% │ 0.5%    │ $1.88    │ $56     │
  │ 第6月    │ 1,200  │ 3.0% │ 0.6%    │ $10.80   │ $108    │
  │ 第12月   │ 4,500  │ 3.5% │ 0.8%    │ $18.00   │ $540    │
  └─────────┴────────┴──────┴─────────┴──────────┴─────────┘

  经常性佣金优先策略（进取）：
  ┌─────────┬────────┬──────────┬──────────┬──────────┐
  │ 时间     │ 日均PV  │ 日均收入  │ 月收入    │ 复利说明  │
  ├─────────┼────────┼──────────┼──────────┼──────────┤
  │ 第3月    │ 300    │ $2.50    │ $75      │ 起步      │
  │ 第6月    │ 1,200  │ $6.67    │ $200     │ 第1-3月客户开始贡献经常性收入 |
  │ 第12月   │ 4,500  │ $26.67   │ $800     │ 第1-9月累积客户的复利效应    │
  └─────────┴────────┴──────────┴──────────┴──────────┘
```

## 3. Non-Goals

| # | 不做什么 | 原因 |
|---|---------|------|
| N1 | **不做全品类折扣聚合** | 调研发现10+成熟平台已占位。折扣信息仅作为辅助模块，不作为核心定位 |
| N2 | **不做全品类软件替代方案（非AI品类）** | 聚焦AI赛道，$1,822亿市场足够大。扩展非AI品类稀释Agent精力且与AlternativeTo正面竞争 |
| N3 | **不做用户注册/登录/收藏/评论** | V1阶段不需要。社区功能在流量 > 10K UV/月再评估 |
| N4 | **不做Newsletter（V1）** | 分散Agent资源。V2阶段（第6月后）评估 |
| N5 | **不做付费工具提交（V1）** | 需要支付网关+审核流程+退款机制，V1信任度不够 |
| N6 | **不做中文主站** | Grill Me + 调研双重验证：中文替代方案搜索量不足以独立支撑 |
| N7 | **不追求一次性覆盖500个替代方案页** | 质量 > 数量。前30个替代页建立质量基准，Agent模仿后扩展 |

## 4. User Stories

### 核心故事（替代方案对比）

| 优先级 | 用户故事 |
|:--:|------|
| P0 | 作为**想换掉当前AI工具的用户**，我希望**在一张对比表里看到10个替代品的价格、核心功能、评分和适合场景**，以便**我5分钟内找到最适合的替代工具，而不是花一下午开10个tab** |
| P0 | 作为**预算有限的独立开发者**，我希望**一眼看出哪些替代品有免费层或更便宜**，以便**优先考虑不超预算的选项** |
| P0 | 作为**搜索引擎用户**，我希望**搜"[tool] alternatives"时直接看到一份结构化对比表**，以便**不需要点进多个网站手动比** |
| P0 | 作为**有特定需求的用户**（如需要API、需要团队协作），我希望**对比表包含功能维度的筛选**，以便**只看到真正符合硬性要求的替代品** |

### 辅助故事（折扣模块）

| 优先级 | 用户故事 |
|:--:|------|
| P1 | 作为**已经选定工具的用户**，我希望**看到这个工具当前是否有折扣**，以便**在最优惠的时候购买** |
| P1 | 作为**预算敏感的用户**，我希望**看到今天有哪些AI工具在打折**，以便**不错过限时优惠** |

## 5. Requirements

### 5.1 替代方案模块（核心，P0）

| # | 需求 | 验收标准 |
|---|------|---------|
| A-P0-1 | **替代方案页**：每个AI工具对应 `/alternatives/[slug]/` | 展示工具简介 + 5-15个替代品结构化对比表（实际数量基于Agent质量评分） |
| A-P0-2 | **对比维度差异化**：Agent根据工具品类自适应选择对比列 | 至少包含：定价模式、最低月价、核心功能（5个tag）、用户评分+来源、API可用性、免费层。品类特征维度（如图像工具含分辨率/风格数） |
| A-P0-3 | **联盟链接自然嵌入**：每个替代品的"Visit"按钮 = 联盟链接 | 跳转使用 `/go/[slug]` 中间页（去平台化+防AdBlock） |
| A-P0-4 | **Schema标记**：FAQ + Product + BreadcrumbList + AggregateRating | Rich Results Test 无报错 |
| A-P0-5 | **内链网络**：替代页之间通过"Related Alternatives"互相链接 | 每个页底部3-5个相关替代页链接 |
| A-P0-6 | **替代方案首页**：`/alternatives/` 按工具流行度列出所有有替代页的工具 | 支持搜索和按品类筛选 |
| A-P0-7 | **空状态处理**：某个工具暂无替代方案 → 显示"Coming soon" + 推荐浏览其他替代方案 | 不显示空白页 |
| A-P0-8 | **替代品质量最低阈值**：Agent判断功能相似度 < 60分的工具不列入对比 | 不为了凑数降低质量 |

## 6. Agent 自动化（V2 修正）

| # | 任务 | 频率 | 关键变化（vs V1） |
|---|------|------|---------|
| AG-1 | **替代方案发现与生成** | 每日 08:00 | **核心任务升级**。扫描PH/HN/Reddit新AI工具 → 用相似度算法匹配已有工具的替代关系 → 生成对比数据。**优先处理有经常性佣金的工具** |
| AG-2 | **对比数据更新** | 每日 10:00 | 抽查10%工具的定价/功能/评分变化 |
| AG-3 | **折扣发现**（辅助） | 每日 09:00 | **降级为辅助任务**。扫描 AppSumo + Reddit r/AIToolsDeals + Twitter AI KOL + 官网pricing page |
| AG-4 | **折扣过期检查** | 每日 11:00 | 标记过期折扣 |
| AG-5 | **内链优化** | 每周日 12:00 | 基于新替代方案的生成，更新Related Alternatives区块 |
| AG-6 | **质量检查** | 每日 14:00 | 死链/Lighthouse/替代品质量分抽查 |
| AG-7 | **变现优化** | 周二/四 16:00 | 联盟链接有效性 + **经常性佣金工具优先级权重调整** |
| AG-8 | **每日构建触发** | 每日 15:00 | 单一构建（所有Agent操作合并触发，< 35次/月） |

### Agent失败降级策略

```
三层保障：
  1. 重试：3次（间隔5/15/30分钟）
  2. 降级：失败后保留昨日数据，不发布空白内容
  3. 告警：Telegram/邮件通知（不静默失败）

数据源冗余：
  - 折扣发现：AppSumo → Reddit → Twitter → 官网（任一可用即可）
  - 替代方案发现：PH → HN → Reddit → 官网（同上）
```

### Agent 数据接口约定（给全栈开发专家）

全栈开发专家需要知道 Agent 将如何与站点交互，以便设计正确的 API/数据层和构建流水线。

```
交互模式：

  Agent（WorkBuddy定时任务）
     │
     │  Supabase REST API / Client SDK
     ▼
  Supabase PostgreSQL
     │  (tools / deals / alternatives 表)
     │
     │  GitHub Actions 每日 15:00 触发
     ▼
  Astro SSG 构建  ──→  Cloudflare Pages 部署
     (读取Supabase数据)
```

| 接口 | 说明 | 约束 |
|------|------|------|
| **Agent → 数据库** | Agent 通过 Supabase Client SDK（JS）或 REST API 直连 Supabase，直接写入 tools / deals / alternatives 表 | 使用 Supabase Service Role Key（服务端安全），不暴露给前端 |
| **数据库 → 构建** | GitHub Actions 每天 15:00 执行 `astro build`，从 Supabase 读取最新数据，生成静态页面 | 所有 Agent 的当日数据操作必须在 15:00 前完成 |
| **构建 → 部署** | `astro build` 产物自动部署到 Cloudflare Pages（通过 Git push 触发或 CF Pages CI） | CF Pages 免费层限制：500次构建/月。Agent 数据写入不直接触发构建，统一通过 GitHub Actions 每日单次触发 |
| **前端 → 联盟跳转** | 所有联盟链接统一走 `/go/[slug]` 路由，由 Astro 在构建时从数据库读取 `affiliate_url` 静态注入 | 不需要服务端运行时，纯静态生成的跳转页 |
| **Agent → 通知** | Agent 异常时通过 Telegram Bot API 或邮件 SMTP 发送告警 | 不依赖站点基础设施，独立运行 |

关键设计约束（全栈开发专家注意事项）：

```
1. Agent 不操作文件系统。所有内容变更通过 Supabase 写入。
   无需为 Agent 提供 CMS、Admin 面板或文件上传接口。

2. 构建是幂等的。每次构建从 Supabase 全量读取 → 生成静态页。
   不存在"增量构建"或"差分更新"需求。

3. Supabase 免费层限制（2026年7月核实）：
   - 500MB 数据库（预估 3,000条工具 × 2KB = 6MB，远低于上限 ✅）
   - 50,000 月活用户（初期足够 ✅）
   - 5GB 出站带宽/月（静态站点从 CF Pages 读取构建产物，几乎不走 Supabase 带宽 ✅）
   - 1GB 文件存储（工具 Logo/截图放 Cloudflare R2，不用 Supabase Storage ✅）
   - 最多 2 个活跃项目
   - 无限 API 请求
   - ⚠️ **7天不活跃自动暂停**：Agent 每日写入数据 = 天然 Keep-Alive，无需额外配置
   - ⚠️ 新项目（2026年5月30日后）需显式 PostgREST 授权才能通过 REST API 访问表

4. Cloudflare Pages 免费层限制：
   - 500 次构建/月 → 日均 1 次 = 30 次/月，安全水位
   - 无限带宽（静态资源通过 CF CDN 全球分发）
   - 单次构建超时 20 分钟 → Astro 构建 500 页预估 5-8 分钟，安全

5. 数据写入速率：Agent 日均写入约 10-30 行
   （3-5个新工具 + 5-10个替代方案关系 + 2-5个折扣），
   Supabase REST API 完全能承载，无需队列或批处理。
```
